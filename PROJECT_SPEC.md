## Overview

Next.js app for a guest-facing wedding invitation site plus an admin dashboard for
managing households/guests, RSVP, and email/SMS communications. Wedding date per
`lib/supabase.ts` defaults: 2027-07-10, QT Hotel Melbourne. Backed by Supabase (Postgres).

- **`app/invite/[slug]/layout.tsx`** (guest invite phases): self-hosted `Fraunces` variable (normal + italic) → `--font-fraunces`; self-hosted `Bricolage Grotesque` variable → `--font-bricolage`; self-hosted `DM Mono` (400/500) → `--font-dm-mono`. - **`app/admin/layout.tsx`**: `next/font/google` `Geist_Mono` and **`Bebas_Neue`** (→ `--font-bebas-neue`) — the one place still fetching from Google Fonts rather than self-hosting.

### Old-font status — Cinzel and Bebas Neue are still live, not legacy

Both are actively used today, scoped to specific surfaces — they have **not** been fully
replaced and shouldn't be treated as dead:
- **Cinzel**: used across `app/admin/**` headings (`GuestListTable.tsx`, `EditHouseholdForm.tsx`, `NewHouseholdForm.tsx`, `TemplatesClient.tsx`) and on the root "find your invitation" page (`app/page.tsx`, `app/invite/[slug]/v3/primitives.tsx`).
- **Bebas Neue**: defined in `app/admin/layout.tsx`, used in `app/admin/comms/templates/page.tsx`.
- `tailwind.config.ts` still maps `bebas-neue`/`cinzel` Tailwind utilities, but this file is **not loaded by Tailwind v4** in this project (no `@config`), so those mappings are dead config superseded by the `@theme inline` block in `globals.css`.

Net: the guest-facing invite experience (v3 landing + v4 phases) has moved to
Fraunces/Bricolage/DM Mono/DM Sans. Cinzel and Bebas Neue remain the real, current
typefaces for the admin dashboard and the homepage — any future "finish the font
migration" work needs to decide whether admin/homepage are in scope, not assume they're
leftover debris.

## Admin — per-guest phase preview modal

`GuestListTable.tsx`: the ↗ icon in the invite-link cell opens a `PreviewPhaseModal` (dark
card, same pattern as `DeleteConfirmModal`) listing all four phases as links for that
specific household. The currently active phase is highlighted in green. Each link opens
`/invite/<slug>?preview=<phase>` in a new tab.

## Admin preview requests and link tracking

`app/invite/[slug]/page.tsx`: `resolvePreviewPhase()` is now called **before**
`getInviteData()` and the result is passed as `isPreview`. When `isPreview` is `true`
the `link_open_count` / `link_first_opened_at` update is skipped entirely. This prevents
admin preview visits from inflating household view counts in the comms dashboard.

## Comms dashboard — "Awaiting RSVP" filter

`CommsSummaryRow` has a `hasPendingRsvp: boolean` field (true when at least one guest
in the household has `rsvp_status` other than `'attending'` or `'declined'`).

`CommsClient` exposes a fifth tab `'awaiting_rsvp'` styled in the admin warning amber.
Filter condition: `row.lastContacted !== null && row.hasPendingRsvp` — i.e. the household
has been contacted at least once but the RSVP is not yet complete for all guests.
